# Nepres Clinic API

Backend MVP для сервиса сопровождения назначенного врачом лечения. Система **не ставит диагнозы и не назначает лечение**: она даёт врачу основу для ведения пациентов, а следующим этапом на её базе появятся цифровые маршруты, ежедневные задания и контроль выполнения.

## Что реализовано в v1

- регистрация новой организации вместе с первым врачом;
- JWT-аутентификация (`Bearer` token), безопасное хранение паролей PBKDF2-SHA256;
- CRUD организаций (только системный администратор);
- системный администратор управляет всеми организациями и врачами, а пациентов создают только лечащие врачи;
- карточка пациента V1: личные данные, канал Magic Link, рост/вес, экстренный контакт, серверный timestamp согласия, диагноз и клинический контекст;
- обычные врачебные аккаунты ограничены своей организацией — их отдельный кабинет будет следующим интерфейсом;
- стартовый системный профиль для будущей админки: **login:** `admin`, **password:** `admin`;
- интерактивная документация OpenAPI: `/docs`;
- pytest-проверки: health-check, bootstrap admin, регистрация, логин, CRUD пациента, разграничение организаций и права администратора.

## Архитектура

Проект организован по Clean Architecture / Onion:

```
app/
  domain/           бизнес-сущности и ошибки, без FastAPI/SQLAlchemy
  application/      use cases, порты репозиториев, JWT и пароли
  infrastructure/   SQLAlchemy-модели, SQLite и реализации репозиториев
  api/              FastAPI-схемы, зависимости, HTTP-маршруты
  main.py           композиция приложения и bootstrap-администратор
```

Зависимости направлены внутрь: HTTP и БД знают о прикладном слое, а домен не зависит от фреймворков. SQLite выбран для быстрого MVP-старта; URL БД вынесен в настройки, поэтому PostgreSQL можно подключить без изменения use cases. Redis для кэша/фоновых сценариев намеренно не добавлен в первую рабочую версию.

## Быстрый старт

Нужен Python 3.11+.

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
uvicorn app.main:app --reload
```

После запуска откройте [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs). База `nepres_clinic.db` и профиль `admin/admin` создаются автоматически при первом старте. Перед публикацией обязательно измените `JWT_SECRET` и пароль администратора.

Нативная админ-панель без React доступна по [http://127.0.0.1:8000/get_admin_page](http://127.0.0.1:8000/get_admin_page). Войдите в неё под `admin` / `admin`; интерфейс использует тот же JWT API и показывает фактические данные из базы.

Отдельный кабинет врача: [http://127.0.0.1:8000/get_doctor_page](http://127.0.0.1:8000/get_doctor_page). В него может войти только неадминистративный аккаунт врача; создание пациента автоматически связывает его с этим врачом.

## Тесты

```bash
pytest -q
```

> На текущем этапе это первая схема V1. Если вы уже запускали раннюю локальную версию до расширения карточки пациента, удалите только локальный файл `nepres_clinic.db` и запустите приложение снова. Перед использованием с реальными данными следующим инфраструктурным шагом будут Alembic-миграции — согласие на обработку данных нельзя корректно проставить задним числом.

## Рабочая цепочка

1. Врач регистрирует организацию через `POST /api/v1/auth/register`.
2. Получает JWT через `POST /api/v1/auth/login` и передаёт его как `Authorization: Bearer <token>`.
3. Врач в дальнейшем работает только со своей организацией через отдельный кабинет.
4. Системный администратор `admin` через админ-панель управляет всеми организациями, врачами и пациентами.
5. Лечащий врач создаёт пациента: запись автоматически связывается с его организацией и профилем врача; системный администратор пациента создать не может.

## Ближайшее развитие

Следующий функциональный модуль: маршруты лечения (`TreatmentPlan`, задания по дням, приемы препаратов, материалы, self-report), публичная пациентская ссылка и события выполнения. После появления фоновых задач стоит подключить Redis для кэша, очередей уведомлений и мониторинга просроченных действий.
